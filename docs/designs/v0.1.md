# claude-shared v0.1 Design

## Goals

1. Extract duplicated Claude Code configuration from Tab42 and tiramisu into a shared, maintainable repo
2. Enable any project to adopt the documentation-first workflow with a single setup command
3. Provide automatic updates so consumers stay current without manual effort

## Design Decisions

### Git submodule as primary distribution

Submodules were chosen over npm packages, symlinks, or copy-paste because:
- Files travel with the repo — collaborators get them on clone
- Version pinning per project — each project can control when to update
- No package registry dependency

### Three installation modes

`setup.sh` supports copy, symlink, and submodule (recommended) to accommodate different environments:
- **Copy**: Air-gapped or fork scenarios where submodules aren't practical
- **Symlink**: Local-only development on a single machine
- **Submodule**: Team collaboration with portable, versioned dependencies

### Hooks resolve git root dynamically

All hooks use `git rev-parse --show-toplevel` instead of relying on `$CLAUDE_PROJECT_DIR` or CWD. This ensures correct behavior when Claude Code is launched from a parent directory but operates on child projects.

### Auto-update at user level

The auto-update hook lives in `~/.claude/settings.json` (not per-project) because:
- It needs to run regardless of which project is active
- It updates the shared repo itself, then syncs project submodules
- At parent directory level, it scans children to update all submodules

### Version tracking

`.claude/.claude-shared-version` records installation mode, commit hash, date, and source URL. This enables:
- Re-run detection (avoid redundant setup)
- Mode switching with cleanup
- Version comparison for manual update checks

## Scope

### In scope for v0.1
- `/document-commit` skill
- `require-commit-doc.sh` pre-commit hook
- `update-design-doc.sh` post-plan hook
- `auto-update.sh` user-level hook
- Interactive `setup.sh` with 3 modes
- Parent-directory hook support

### Future considerations
- Additional skills (e.g. `/review-pr`, `/changelog`)
- Plugin packaging for broader distribution

### LLM-wiki (planned)

Each project gets a living wiki at `docs/wiki/` — organized by topic, written in present tense, describing what is true **right now**. Unlike version docs (commit snapshots) or design docs (version-scoped plans), the wiki is the canonical source of truth that Claude reads before coding and updates as it goes.

**Key concepts:**
- **Read-before-act**: Claude reads relevant wiki pages before making code changes, so it always has architectural context
- **Continuous updates**: Wiki updates happen during development, not just at commit or plan boundaries — when decisions are made, patterns introduced, or assumptions change
- **Present tense, code-referenced**: Wiki content says "the API uses JWT" (not "we decided to use JWT"), with specific file/line references
- **Self-correcting**: If wiki contradicts code, code wins — Claude updates the wiki to match reality
- **Topic-based pages**: `overview.md`, `architecture.md`, `data-model.md`, `decisions.md`, `open-questions.md`, etc. — split when pages grow beyond ~100 lines

**Planned components:**
- `/update-wiki` skill — conversational wiki editing with codebase verification
- `wiki-aware.md` rule — scoped to source files, tells Claude to check the wiki before editing code
- `/scaffold-docs` integration — generates initial wiki pages from codebase analysis instead of static templates

**Shift from current model:** docs go from being a byproduct of commits to being the interface between Claude sessions — the primary way knowledge transfers across conversations.
